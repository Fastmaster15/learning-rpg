# Large Field Map Scale Addendum

## 目的

`2026-05-11-field-battle-visual-reference-and-modern-hero.md` および `2026-05-11-field-tile-granularity-and-hero-layer-addendum.md` のフィールド方針を上書きする補足タスク。

以前のタスク内には「既存5x5を維持する」「5x5論理マップをベースにする」という安全寄りの記述があるが、これは次フェーズの方針としては弱い。

ユーザー要望により、Learning RPG のフィールドは **もっと広く、壮大に、冒険している感覚が出るもの** を目指す。

---

## 1. 方針変更

### 1.1 旧方針

```txt
既存5x5マップを維持する
5x5のまま見た目だけ細かくする
```

### 1.2 新方針

```txt
5x5は旧プロトタイプ扱いとする。
次フェーズでは、より大きいフィールドマップを前提に設計する。
最低でも 15x15、可能なら 21x21 以上のフィールドを目指す。
```

---

## 2. 目指すフィールド体験

フィールドに出た瞬間、プレイヤーに以下を感じさせたい。

- 世界が広い。
- 町の外に本当に冒険が広がっている。
- 道を進むと、草原、森、山、水辺、目的地が連続して現れる。
- 町の周囲は安全だが、遠くへ行くほど危険そうに見える。
- 宝箱や小目的地を探したくなる。
- 画面外にも世界が続いていそうに見える。
- 主人公が広い世界の中を歩いている。

重要なのは「25マスの盤面」ではなく、**小さな世界地図を探索している感覚**。

---

## 3. 推奨マップサイズ

### 3.1 P0 推奨

```txt
15x15
```

理由：

- 既存5x5より明確に広く感じる。
- スマホでもズーム・スクロール・ビューポート設計で扱える。
- 町、草原、森、宝箱、小ボス、川、山道を配置できる。
- 実装負荷と体験改善のバランスがよい。

### 3.2 P1 推奨

```txt
21x21
```

理由：

- かなりフィールドらしくなる。
- 町からボスまでに中間地点を置ける。
- 地形の変化、道の分岐、探索の余白を作れる。
- 今後の章追加にも耐えやすい。

### 3.3 P2 推奨

```txt
32x32 以上
```

理由：

- 本格的なワールドマップに近づく。
- ただし、移動テンポ、エンカウント、セーブ、ミニマップなど追加設計が必要。
- 今回のP0では無理に狙わなくてよい。

## 4. ビューポート設計

マップ全体が大きくなるため、画面には全体を常時表示しなくてよい。

### 4.1 推奨方式

```txt
大きなフィールドマップを持つ
画面には主人公周辺の一部だけを表示する
主人公が移動すると表示範囲が追従する
```

例：

- 内部マップ: 21x21
- 表示範囲: 7x7 または 9x9
- 主人公: 可能な限り表示範囲の中央付近

### 4.2 スマホ表示

スマホでは以下を推奨する。

```txt
表示ビューポート: 7x7
1タイル: 40〜48px程度
主人公は中央付近に固定
マップの方が動くように見せる
```

### 4.3 PC表示

PCでは以下も可。

```txt
表示ビューポート: 9x9 または 11x11
サイドにステータス・ログ・ミニマップを表示
```

## 6. 壮大さを出す地形配置

### 6.1 町周辺

- 町の周囲は grass / road を中心にする。
- 初心者向けの安全地帯に見せる。
- 町から道が外へ伸びていること。

### 6.2 草原

- 町周辺から広がる基本地形。
- 敵は弱め。
- 最初のレベリング場所。

### 6.3 森

- 草原の外側に配置する。
- 森に入ると少し危険そうに見える。
- 宝箱や小目的地を置く。

### 6.4 深い森

- ボスや重要イベントの手前に置く。
- 通常の森より暗く、密度を高くする。
- エンカウントも強めにできる。

### 6.5 山・丘

- マップの端や行き止まりに置く。
- 世界の境界、進行制御、地形の変化に使う。

### 6.6 水辺・海岸

- マップ端や川として配置する。
- 通れない場所として、自然な境界に使う。
- 橋を置くと、進行導線として使える。

### 6.7 道

- 町から主要目的地まで伸ばす。
- 道から外れると探索・宝箱・敵遭遇がある。
- 道はプレイヤーの心理的導線になる。

## 7. 主人公の表示方針

大きなフィールドでも、主人公は現在地タイルの上に乗って見えること。

### 7.1 表示方式

推奨：

```txt
マップ全体: 大きい
表示範囲: 主人公周辺だけ切り出す
主人公: 表示範囲中央のタイル上にオーバーレイ
```

つまり、主人公がタイルを置き換えるのではなく、地形の上に立つ。

### 7.2 中央固定方式

可能なら以下を採用する。

```txt
主人公は画面中央付近に固定
移動すると表示するマップ範囲が変わる
```

これにより、広いマップを歩いている感覚が出る。

### 7.3 端の処理

マップ端では主人公を必ずしも中央固定にしなくてよい。

- マップ左端では、表示範囲が左端に固定される。
- 主人公は画面中央より左に寄ってもよい。
- ただし、現在地は常に分かること。

## 8. カメラ / 表示範囲ロジック

### 8.1 基本ロジック

```ts
const VIEWPORT_SIZE = 7; // or 9
const half = Math.floor(VIEWPORT_SIZE / 2);

const startX = clamp(player.x - half, 0, mapWidth - VIEWPORT_SIZE);
const startY = clamp(player.y - half, 0, mapHeight - VIEWPORT_SIZE);

const visibleTiles = fieldMap
  .slice(startY, startY + VIEWPORT_SIZE)
  .map(row => row.slice(startX, startX + VIEWPORT_SIZE));
```

### 8.2 HeroSprite表示位置

```ts
const heroViewportX = player.x - startX;
const heroViewportY = player.y - startY;
```

この `heroViewportX/Y` のタイル上に `HeroSprite` をオーバーレイする。

## 9. P0実装要件

今回のP0では、以下を最低限達成する。

- 5x5固定ではなく、最低15x15のフィールドデータを持つ。
- 表示範囲は7x7または9x9に切り出す。
- 主人公は現在地タイルの上にオーバーレイ表示する。
- 町、草原、森、深い森、道、水辺、山、宝箱、ボス地点を配置する。
- 移動可能 / 不可能の判定が維持される。
- 町に戻れる。
- 宝箱を取得できる。
- ボス地点へ到達できる。
- エンカウントが広いマップでも成立する。
- 画面上で「世界が広い」ことが伝わる。

## 10. エンカウント調整

マップが広くなるため、エンカウント頻度を調整する。

### 10.1 基本方針

- 町周辺は低エンカウント。
- 草原は通常。
- 森はやや高め。
- 深い森は高め。
- 道は低め。
- 水・山は基本的に通行不可。

### 10.2 タイル別例

| タイル | エンカウント |
|---|---:|
| road | 低い |
| grass | 普通 |
| forest | やや高い |
| deep_forest | 高い |
| town | なし |
| water | なし / 通行不可 |
| mountain | なし / 通行不可 |
| boss | 固定イベント |
