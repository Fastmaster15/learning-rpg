# Classic JRPG Experience Requirements

## 目的

現在のRPGプロトタイプは、まだ「遊べるRPG体験」としては初期段階にある。
次の開発では、見た目や演出の作り込みよりも、プレイヤーが気持ちよく前に進める基本体験を成立させることを最優先にする。

この要件書は、機能一覧ではなく、**プレイヤーがどう気持ちよく進むか**を中心に設計するためのものとする。

## 目指す体験

本作は、クラシックなコマンド選択式JRPGの基本体験を持つブラウザRPGを目指す。

重視するのは、既存有名RPGの固有名詞・UI・音楽・キャラクターを模倣することではなく、以下のような体験構造である。

- 町で情報を集める
- フィールドへ出る
- 敵と遭遇する
- コマンドで戦う
- 経験値とゴールドを得る
- レベルアップや装備更新で強くなる
- 町に戻って回復する
- 宝箱や小目的地で探索の報酬を得る
- 小ボスを倒して一区切りの達成感を得る
- もう少し遠くへ行きたくなる

派手さよりも、シンプルで分かりやすく、少しずつ前に進みたくなるテンポを優先する。

---

## 今回の開発ゴール

プレイヤーが5〜10分間、明確な目的を持って遊べる状態にする。

最低限、以下の流れを自然に体験できること。

1. タイトル画面からゲームを開始する
2. 町にいる
3. NPCと話して目的を知る
4. フィールドに出る
5. 敵と遭遇する
6. コマンド式バトルで戦う
7. 勝利して経験値とゴールドを得る
8. レベルアップする
9. HP/MPが減る
10. 町に戻って回復する
11. ゴールドで装備を更新する
12. 宝箱または探索報酬を見つける
13. 強くなった状態で森へ進む
14. 小ボスを倒す
15. 最初の小目標を達成する
16. 達成メッセージや勝利音によって一区切り感を得る

### プレイヤーに感じてほしいこと

- 最初は弱い
- 何度か戦うと少し強くなる
- 装備を買うと戦闘が楽になる
- 無理をすると負ける
- 町に戻ると安心できる
- 宝箱を見つけると探索してよかったと思える
- 小ボスを倒すと一区切りの達成感がある
- 勝利音やレベルアップ音で成果を直感的に感じる
- 次の目的地へ進みたくなる

### 重要な体験キーワード

- 安心できる町
- 少し怖いフィールド
- 分かりやすい戦闘
- 数値で分かる成長
- 町に戻る判断
- 探索の小さな報酬
- 節目としての小ボス
- 音で感じる達成感
- 次の目的地への誘導

- 敵を倒すメリットがある

- ゴールドを使う先がある
- レベルアップで明確に強くなる
- 探索すると報酬がある
- 小ボスを倒すことで一区切りの達成感がある
- 次に行く場所が分かる

## 必須画面

### 1. タイトル画面

目的：ゲーム開始前の入口を作る。

要件：

- ゲームタイトルを表示する
- 「はじめる」ボタンを表示する
- セーブ機能がある場合のみ「つづきから」を表示する
- まずは「はじめる」のみでもよい

完了条件：

- ボタンを押すと町画面へ遷移する

### 2. 町画面

目的：プレイヤーの安全地帯・拠点にする。

要件：

- NPCが最低3人いる
- 宿屋または回復ポイントがある
- 武器屋または装備更新ポイントがある
- フィールドへ移動できる出口がある
- プレイヤーのHP/MP/GOLDを確認できる

町に必要な要素：

- NPC A：目的地を教える
- NPC B：戦闘や回復のヒントを教える
- NPC C：世界観や雰囲気を伝える
- 宿屋：HP/MPを全回復する
- 店：武器または防具を買える
- 出口：フィールドへ移動する

完了条件：

- NPCに話しかけられる
- 宿屋で回復できる
- 店で装備を買える
- フィールドへ移動できる

### 3. フィールド画面

目的：探索と戦闘の場にする。

要件：

- プレイヤーが上下左右に移動できる
- 一定歩数または確率で敵と遭遇する
- 町に戻れる
- 最初の目的地が存在する
- 宝箱が少なくとも1つある
- 小ボスのいる場所がある
- HPが減った状態で探索を続けるリスクがある

フィールド要素：

- 町への入口
- 草原エリア
- 森エリア
- 小目的地
- 宝箱
- 小ボス地点
- 敵エンカウント

完了条件：

- 移動できる
- 敵と遭遇する
- 宝箱を見つけられる
- 町に戻れる
- 目的地に到達できる
- 小ボスに挑める

### 4. 戦闘画面

目的：クラシックJRPGらしいコマンド戦闘を成立させる。

要件：

- プレイヤーと敵の状態を表示する
- コマンドを選択できる
- 戦闘ログを表示する
- 勝敗がある
- 勝利時に報酬がある
- 敗北時に町へ戻される
- 小ボス戦では通常戦闘よりも緊張感を出す

表示項目：

- プレイヤーHP/MP
- 敵名
- 敵HP
- コマンド
- 戦闘ログ

コマンド：

- たたかう
- じゅもん
- どうぐ
- にげる

完了条件：

- コマンドで戦える
- 敵を倒せる
- 敵からダメージを受ける
- 勝利報酬を得られる
- 小ボスを倒せる
- 敗北時に町へ戻る

### 5. ステータス画面

目的：成長を見える化する。

要件：

- プレイヤーの現在ステータスを表示する
- 装備を表示する
- 次のレベルまでの経験値を表示する

表示項目：

- 名前
- レベル
- HP
- MP
- 攻撃力
- 防御力
- 経験値
- 次のレベルまで
- ゴールド
- 武器
- 防具

完了条件：

- 現在の強さが分かる
- レベルアップ後に数値変化が分かる

### 成長方針

- レベル1では弱い敵にも少し苦戦する
- レベル2になると弱い敵が楽になる
- レベル3になると普通の敵にも安定して勝てる
- レベル2〜3かつ装備更新後に小ボスへ挑める
- 装備を更新すると明確に被ダメージや与ダメージが変わる

### レベルテーブルの目安

- maxHp +5〜8
- maxMp +2〜4
- attack +2〜3
- defense +1〜2
- HP/MPを全回復してもよい
- レベルアップログまたは効果音を出す

## 戦闘仕様

### 基本ルール

- ターン制
- プレイヤーが先に行動する
- プレイヤー行動後、敵が生きていれば敵が行動する
- どちらかのHPが0になるまで繰り返す

### ダメージ計算

```ts
damage = Math.max(1, attacker.attack - defender.defense + random(-2, 2));
```

要件：

- 最低1ダメージは入る
- ランダム幅を少し入れる
- クリティカルは後回しでよい

### じゅもん

最初は2種類でよい。

```ts
const spells = [
  {
    id: "fire",
    name: "火の玉",
    mpCost: 3,
    effect: "enemy_damage",
    power: 12
  },
  {
    id: "heal",
    name: "小回復",
    mpCost: 3,
    effect: "self_heal",
    power: 18
  }
];
```

要件：

- MPが足りない場合は使えない
- 攻撃魔法は通常攻撃より強い
- 回復魔法は探索継続の判断材料になる

### どうぐ

初期アイテム：

```ts
const items = [
  {
    id: "herb",
    name: "薬草",
    effect: "heal",
    power: 20,
    price: 10
  }
];
```

要件：

- 薬草を使うとHPが回復する
- 使うと個数が減る
- 個数が0なら使えない

### にげる

要件：

- 成功率は70%
- 成功時はフィールドに戻る
- 失敗時は敵の攻撃を受ける
- 小ボス戦では逃走不可、または逃走成功時に森の入口へ戻す

### 戦闘ログ

ログは短文で表示する。

例：

- Heroのこうげき！
- 敵に5のダメージ！
- 敵のこうげき！
- Heroは3のダメージをうけた！
- 敵をたおした！
- 経験値4を手に入れた！
- 8ゴールドを手に入れた！
- レベルが上がった！
- 小ボスをたおした！

### 勝利処理

- 経験値を得る
- ゴールドを得る
- レベルアップ判定をする
- 勝利ログを表示する
- 可能であれば勝利音を鳴らす
- 通常戦闘ではフィールドへ戻る
- 小ボス戦ではクエスト進行フラグを更新する

### 敗北処理

- 町に戻る
- HPを最大値の半分まで回復する
- ゴールドを減らす処理は今回は不要
- 「目を覚ますと町に戻っていた」などの簡単なログを表示する

## 小ボス仕様

### 目的

最初のミニクエストに明確な締めを作り、プレイヤーに「準備して挑み、勝った」という達成感を与える。

### 小ボスの役割

- 雑魚戦で得た経験値とゴールドを使う理由を作る
- レベルアップと装備更新の効果を確認させる
- 最初のクエストクリアに節目を与える
- プロトタイプに一区切りのゴールを作る

### 小ボス要件

- 小ボスは森の奥に配置する
- レベル1では勝ちにくい強さにする
- レベル2〜3、かつ装備更新後に安定して勝てる強さにする
- 小ボス戦の前に、NPCまたはフィールド上のヒントで危険を伝える
- 小ボス勝利後、達成メッセージを表示する
- 小ボス勝利後、通常より大きい経験値とゴールドを与える
- 可能であれば勝利ジングルまたは特別な勝利ログを出す

### 小ボス勝利後メッセージ例

> 森のぬしをたおした。
> 北の森の光は、静かに消えていった。
> 小さな冒険者として、一歩を踏み出した。

### 完了条件

- プレイヤーが雑魚戦で準備する意味を感じる
- 小ボスに勝つことでミニクエストが完了する
- クリア後に「ここまで遊んだ」という達成感がある

## 町・NPC仕様

NPCは単なる飾りではなく、プレイヤーの行動を誘導する。

### NPC A：目的提示

例：

> 北の森で、あやしい光を見た人がいるらしい。

目的：次の目的地を示す。

### NPC B：戦闘ヒント

例：

> 傷ついたら無理をするな。町に戻って休むんだ。

目的：回復の重要性を伝える。

### NPC C：装備ヒント

例：

> 森に行くなら、少し良い武器を持っていくと安心だよ。

目的：ゴールドを使う理由を伝える。

### NPC D：小ボス警告

例：

> 森の奥には、普通の魔物とは違う気配がある。準備してから行くんだ。

目的：小ボスに向けた準備行動を促す。

### NPC E：世界観

例：

> この町は旅人たちの休み場さ。外は魔物が多いけどね。

目的：世界観を少しだけ出す。

### NPC会話要件

- 1会話は1〜3文まで
- 1文は短くする
- 次に何をすればいいか分かる内容を優先する
- 難解な設定説明は避ける
- 小ボス前には「準備が必要」と分かるヒントを出す

## フィールド・探索仕様

### 移動

- 上下左右に移動できる
- キーボードまたは画面ボタンで操作できる
- スマホ対応はP1でよいが、最低限クリック・タップ操作も考慮する

### エンカウント

方式：

- 一定歩数ごとに確率判定する
- 例：5歩以降、1歩ごとに20%でエンカウント

```ts
if (stepsSinceBattle >= 5 && Math.random() < 0.2) {
  startBattle();
}
```

### エリア

最初は2エリアでよい。

#### 草原

- 町の近く
- 弱い敵が出る
- 初心者向け

#### 森

- 少し遠い
- 強めの敵が出る
- 宝箱がある
- 小ボス地点がある
- 小目的地がある

## 宝箱仕様

### 目的

探索する意味を作り、フィールド移動に小さな報酬を与える。

### 要件

- 最初のプロトタイプでは宝箱は1つでよい
- 宝箱は森または草原の少し奥に置く
- 中身は薬草、ゴールド、簡単な装備のいずれかにする
- 一度開けた宝箱は再取得できない
- 宝箱を開けたときは短いログを表示する
- 可能であれば宝箱オープン音を鳴らす

### 初期宝箱の目安

```ts
const treasureChests = [
  {
    id: "forest_chest_001",
    area: "forest",
    rewardType: "item",
    rewardId: "herb",
    rewardCount: 2,
    opened: false
  }
];
```

### 宝箱ログ例

- 宝箱を見つけた！
- 薬草を2つ手に入れた！

### 完了条件

- プレイヤーが移動中に宝箱を発見できる
- 宝箱の中身を取得できる
- 宝箱を開けたことがログで分かる
- 探索してよかったという小さな報酬感がある

## 最初のミニクエスト

### クエスト名

北の森の光

### 目的

プレイヤーに最初の明確なゴールを与える。

### 流れ

1. 町のNPCが北の森の異変を話す
2. プレイヤーがフィールドに出る
3. 草原で敵と戦って成長する
4. 宝箱を見つけて小さな報酬を得る
5. ゴールドを貯めて装備を更新する
6. 森に進む
7. 森の奥で小ボスと戦う
8. 小ボスに勝利後、光を調べる
9. 町に戻る
10. NPCから感謝される
11. 勝利音または達成メッセージを出す
12. 「プロトタイプはここまで」と表示する

### クリア条件

- 森の奥のイベントを確認する
- または小ボスを倒す

### 報酬

- 経験値
- ゴールド
- 簡単な称号またはメッセージ

例：

> 小さな冒険者として、一歩を踏み出した。

## UI / UX方針

### 基本方針

- 画面はシンプルにする
- 次に押せるボタンを分かりやすくする
- 状態変化を必ず見せる
- ログを使って行動結果を明確に伝える
- 音は補助として使い、音なしでも遊べる状態にする

### 常時表示したい情報

- HP
- MP
- Lv
- Gold
- 現在地

### 戦闘UI

優先順位：

1. コマンドが押しやすい
2. HP変化が分かる
3. ログが読みやすい

## 音楽・効果音仕様

### 目的

プレイヤーの達成感、緊張感、安心感を補助する。
音はゲーム体験を強めるが、実装初期では音なしでも遊べる構造を優先する。

### 著作権・表現方針

- 既存有名RPGの音楽、効果音、メロディ、ジングルは直接使わない
- 既存有名RPGに似せすぎない
- 独自のレトロRPG風、またはライセンス上安全な素材のみ使用する
- 音源の出典やライセンスはREADMEまたは開発メモに記録する

### P0：最小効果音

可能であれば、以下を優先して入れる。

- 勝利音
- レベルアップ音
- 宝箱を開けた音
- 小ボス勝利時の短いジングル

P0の音は、BGMよりも「成果が出た瞬間」を強調するために使う。

### P1：BGM

余裕があれば、以下を追加する。

- 町BGM：安心感
- フィールドBGM：探索感
- 戦闘BGM：緊張感
- 小ボスBGM：通常戦闘より少し強い緊張感

### 実装方針

- 音量は控えめにする
- ミュート設定を用意できるとよい
- 音声ファイルは `public/audio/` などに分離する
- 音が再生できない環境でもゲーム進行に影響しないようにする

### 完了条件

- 勝利、レベルアップ、宝箱、小ボス勝利のいずれかで音が鳴る
- 音がなくてもログや画面表示で結果が分かる
- 使用素材のライセンスが安全である

## バランス調整方針

### プレイ時間

最初のプロトタイプは5〜10分で一区切りつくようにする。

### レベルアップ頻度

- 最初の2〜3戦でレベル2になる
- さらに3〜5戦でレベル3になる
- レベルアップの喜びを早めに出す
- 小ボス前にレベル2〜3へ到達できるようにする

### ゴールド

- 弱い敵を5〜7体倒すと最初の装備が買える
- 装備更新後、戦闘が少し楽になる
- 小ボス前に装備更新の選択肢が現実的になるようにする

### 宝箱

- 宝箱は攻略を必須にしない
- 取ると少し楽になる程度の報酬にする
- 探索の意味を作るが、取れなくても詰まないようにする

### 戦闘難易度

- 町の近くでは負けにくい
- 森では油断すると負ける
- 回復や装備更新をすると安定する
- 小ボスは準備なしでは厳しいが、準備後は勝てる
